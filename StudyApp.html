⚙️ Admin Basics

Salesforce Fundamentals
QWhat is CRM and what is Salesforce? ▼
CRM (Customer Relationship Management) is software that helps organizations manage customer data, track interactions, and improve relationships.

Salesforce CRM is a cloud-based CRM platform. Users log in via username/password. It includes built-in apps like Sales Cloud and Service Cloud, plus features like Flows, Approval Processes, Validation Rules, Formulas, and Web-to-Lead.

💡 Key point: Salesforce is also a PaaS (Platform as a Service) — you can build custom apps on top of it.
QDifference between Sales Cloud and Service Cloud? ▼
Sales CloudService Cloud
Manages sales operationsManages customer service / help desk
Objects: Leads, Accounts, Contacts, OpportunitiesObjects: Cases, Accounts, Contacts
Features: Web-to-Lead, Lead Assignment Rules, Sales ProcessesFeatures: Omni-Channel, Case Assignment Rules, Service Processes
QWhat are the key Standard Objects and their purpose? ▼
ObjectPurpose
LeadPotential customer before qualification (Sales Cloud)
AccountA company or organization you do business with
ContactIndividual person linked to an Account
OpportunityA potential sales deal / revenue pipeline (Sales Cloud)
CaseCustomer service request or complaint (Service Cloud)
UserPerson who uses Salesforce; stores employee details and controls access
QWhat is a Profile? ▼
A Profile controls what a user can do in Salesforce. It defines:

  • Object-level access (Create, Read, Edit, Delete)
  • Field-Level Security (FLS)
  • Apps and Tabs visible
  • Login IP ranges and hours
Examples: Standard User, System Administrator, Chatter Free
Tab permissions: Default On | Default Off | Tab Hidden
QWhat is a Permission Set vs Permission Set Group? ▼
Permission Set: Grants additional permissions to a user without changing their profile. Used to extend access for specific users.

Permission Set Group: A collection of multiple Permission Sets bundled together for easier bulk assignment.

💡 Think of Profile as the baseline and Permission Sets as top-ups.
QWhat is a Record Type? When is it used? ▼
Record Type allows you to:
  • Display different picklist options based on profile
  • Display different page layouts based on profile
Example: To display different Opportunity Stage values for different sales teams → use Record Type + Sales Processes.
QWhat happens when a Lead is converted? ▼
When a Lead is converted:

  • ✅ An Account is created
  • ✅ A Contact is created
  • ✅ An Opportunity is optionally created
Converted leads are NOT visible in the Lead tab — they can only be accessed via reports.
QWhat are Reports and Dashboards? Types? ▼
Report Formats:
  • Tabular – Simple rows and columns (like a spreadsheet)
  • Summary – Groups of data with subtotals
  • Matrix – Groups rows AND columns
  • Joined – Multiple report blocks in one

Dashboard: Visual representation of one or more reports. Max 20 components.

Dynamic Dashboard: Shows data based on the viewer's permissions — set via "Runs as the logged-in user (Dashboard Viewer)".
⚠️ Report Type ≠ Report Format. Report Type defines which objects/fields are available.
QData Import Wizard vs Data Loader – differences? ▼
FeatureData Import WizardData Loader
InterfaceBrowser-based UIDesktop application
Record LimitUp to 50,000Up to 5 million
ObjectsLimited standard + all customAlmost all objects
OperationsInsert, Update, UpsertInsert, Update, Upsert, Delete, Export
QWhat is a Validation Rule? Give an example. ▼
A Validation Rule ensures data quality. It evaluates a formula — if it returns TRUE, the rule fires and shows an error message.

Examples:
/* Close date cannot be in future */ CloseDate > TODAY() /* Required picklist value */ ISPICKVAL(StageName, "Closed Won") /* Field must not be blank */ ISBLANK(Text_Field__c)

🔐 Security & Access

Salesforce Security Model (4 Layers)
QWhat is OWD (Organization-Wide Defaults)? ▼
OWD defines the baseline access level for records a user doesn't own. It's the most restrictive layer.

Options:
  • Private – Only record owner and above in hierarchy
  • Public Read Only – Everyone can see, only owner can edit
  • Public Read/Write – Everyone can see and edit
  • Public Read/Write/Transfer – Plus can transfer ownership
  • Controlled by Parent – Child follows parent's sharing
💡 Always set OWD as restrictive as possible, then open up with Role Hierarchy and Sharing Rules.
QWhat is Role Hierarchy? ▼
Role Hierarchy gives users higher in the hierarchy access to records owned by users below them.

Can be disabled for custom objects using the "Grant Access Using Hierarchy" checkbox in Sharing Settings.
Important: Role ≠ Profile. Profile controls what you can do. Role controls what records you can see.
QWhat are Sharing Rules? When are they used? ▼
Sharing Rules grant access to records that are restricted by OWD. They can share based on:
  • Record Owner – Who owns the record
  • Criteria – Field values on the record
You can share with: Public Groups, Roles, or Roles and Subordinates.
QQueues vs Public Groups – what's the difference? ▼
QueuesPublic Groups
Used for load balancingUsed in sharing rules, folder access
Can own records (Lead, Case, Custom objects)Cannot own records
Visible as record owners in list viewsUsed in email alerts and sharing rules
QView All vs Modify All — difference? ▼
View All – Regardless of sharing settings, user can see all records of that object (overrides OWD).

Modify All – Regardless of sharing settings, user can see, edit, delete, and transfer all records of that object.

These are set at the Profile level and bypass the entire sharing model.
QWhat is a Restriction Rule? ▼
A Restriction Rule filters records based on specific criteria, making access more restrictive (not more permissive). It works on top of existing sharing settings.

Use case: Even if OWD is Public Read/Write, a Restriction Rule can hide records that don't match a user's criteria.
QWhat is a Lookup Filter? ▼
A Lookup Filter restricts the list of records available in a lookup field. It ensures users only see relevant records when selecting from a lookup.

Example: On the Case object, filter the Contact lookup to only show Contacts linked to the selected Account.

🗃️ Data Model & Relationships

Object Relationships
QLookup vs Master-Detail — core differences? ▼
FeatureLookupMaster-Detail
CouplingLoosely coupledStrongly coupled
Child without parent?✅ Yes, child can exist alone❌ No, parent is required
Cascade DeleteNo✅ Yes — delete master deletes all children
SharingParent & child can have different ownersChild sharing is controlled by parent
Roll-Up Summary❌ Not available✅ Available on master object
QHow to convert Lookup to Master-Detail? ▼
Steps:
  • 1. Ensure all child records have a parent (no blank lookup fields)
  • 2. Go to Object Manager → Fields & Relationships
  • 3. Edit the Lookup field → Change type to Master-Detail
Salesforce will only allow conversion if no child records have an empty parent field.
QWhat is a Junction Object and when do you use it? ▼
A Junction Object creates a many-to-many (M:M) relationship between two objects using two Master-Detail relationships.

Example: A Student can enroll in many Courses. A Course can have many Students.
→ Junction Object: Enrollment (with M-D to Student and M-D to Course)

For EcoConnect: Volunteer ↔ Initiative via a Volunteer_Assignment__c junction object.
QWhat is a Roll-Up Summary Field? ▼
A Roll-Up Summary field is created on the master object in a Master-Detail relationship. It aggregates data from child records.

Functions available: COUNT, SUM, MIN, MAX

Example: On the Account (master), a Roll-Up Summary counts the total number of Cases (children).
QWhat is a Cross-Object Formula? ▼
A Cross-Object Formula references fields from a parent object in a formula field on the child.

Example: On the Contact object, a formula field that shows Account.Industry — this is a cross-object formula referencing the parent Account.
QWhat is an External ID? What field types support it? ▼
An External ID is a field that stores a unique identifier from an external system (ERP, database). Used for:
  • Integrating Salesforce with other systems
  • Upsert operations (update if exists, insert if not)
  • Searching records using external values
Supported field types: Text, Number, Email
Check the "External ID" checkbox when creating the custom field.
QCustom Settings vs Custom Metadata — when to use each? ▼
FeatureCustom SettingsCustom Metadata
Deployable?❌ Data not included in deployments✅ Records deploy with metadata
Accessible in Formulas?Limited✅ Yes
TypesList, HierarchySingle type, many records
Use caseApp config data per user/profileConfig that must travel between orgs
💡 Prefer Custom Metadata for config data you need to deploy (e.g., tax rates, routing rules for EcoConnect).
QWhat are all the relationship types in Salesforce? ▼
  • Lookup – Loose; child can exist without parent
  • Master-Detail – Tight; child depends on parent; supports rollups
  • Self-Relationship – Lookup to the same object (e.g., Account → Account for parent accounts)
  • Hierarchical – Lookup on the User object to itself (e.g., Manager field)
  • Many-to-Many – Via a Junction Object (two M-D relationships)
  • External Lookup – Links to an external data source using external ID
  • Indirect Lookup – Links standard/custom object to an external object using a unique key

⚡ Automation & Flows

Salesforce Automation Tools
QWhat is a Flow? What are the types? ▼
Flow is Salesforce's declarative automation tool to automate repeated business activities.

Types:
  • Screen Flow – Interactive UI for users (like a wizard)
  • Record-Triggered Flow – Runs automatically when a record is created/updated/deleted
  • Auto-Launched Flow – Runs in background, no UI
  • Scheduled Flow – Runs at a specific time
  • Platform Event-Triggered Flow – Triggered by platform events
QWhat context does a Record-Triggered Flow run in? ▼
Record-Triggered Flows run in System Context by default — meaning they have access to all records and bypass sharing rules.

This is why you must be careful about what data operations you perform in a Record-Triggered Flow — users may not intend to share data that the flow touches.
QWhat are the ways to invoke/trigger a Flow? ▼
Flows can be invoked via:
  • 🔘 Button on a record page
  • 💻 Apex code (programmatically)
  • 🔧 Process Builder (legacy)
  • 🔀 Subflow (from another Flow)
  • 📝 Record Changes (Record-Triggered Flow)
  • 📅 Schedule (Scheduled Flow)
QWhat is Field Dependency? ▼
Field Dependency links a Controlling Field to a Dependent Picklist. The values available in the dependent picklist change based on what's selected in the controlling field.

Example in EcoConnect: If Issue Type = "Pollution", then Sub-Type picklist shows "Air", "Water", "Soil". If Issue Type = "Dumping", it shows different options.
QHow do you auto-assign Leads or Cases? ▼
Use Assignment Rules:
  • Go to Setup → Lead Assignment Rules (or Case Assignment Rules)
  • Define criteria (e.g., Region = "Chennai")
  • Assign to a specific User or Queue
Only one Assignment Rule can be active at a time per object.
QWhat are Auto-Launched Flows? ▼
Auto-Launched Flows run in the background with no screen interactions (no UI elements). They can be triggered:
  • From Apex code
  • From another Flow as a subflow
  • From a REST API call
  • From Process Builder (legacy)

🔄 Apex Triggers

Core Trigger Concepts
QWhat is the difference between Trigger.new and Trigger.old? ▼
VariableContainsAvailable in
Trigger.newNew/updated version of recordsinsert, update, undelete
Trigger.oldPrevious version of recordsupdate, delete
Trigger.newMapMap of Id → new recordupdate (before & after), after insert
Trigger.oldMapMap of Id → old recordupdate, delete
💡 Use Maps when you need to look up records by ID efficiently — avoids SOQL inside loops.
QBefore trigger vs After trigger — key difference? ▼
Before Trigger:
  • Fires before the record is saved to the database
  • You can modify Trigger.new fields directly without a separate DML statement
  • Use for: validation, field updates before save
After Trigger:
  • Fires after the record is saved — the record now has an ID
  • Cannot modify Trigger.new — need a separate DML (update) to change the record
  • Use for: related record creation, callouts
QWhat is Bulkification? Why does it matter? ▼
Bulkification means writing trigger code that can handle 200 records at once efficiently (the max batch size Salesforce sends to a trigger).

Bad pattern (NOT bulkified):
for(Case c : Trigger.new) { // SOQL inside loop = governor limit hit! Account a = [SELECT Id FROM Account WHERE Id = :c.AccountId]; }
Good pattern (bulkified):
// Collect all IDs first Set<Id> accIds = new Set<Id>(); for(Case c : Trigger.new) { accIds.add(c.AccountId); } // One SOQL outside the loop Map<Id, Account> accMap = new Map<Id, Account>( [SELECT Id, Name FROM Account WHERE Id IN :accIds] );
QWhat is a Recursive Trigger? How do you prevent it? ▼
A Recursive Trigger is when a trigger fires, makes a DML operation, which fires the trigger again — creating an infinite loop.

Prevention: Use a static Boolean variable
public class TriggerHelper { public static Boolean hasRun = false; } // In trigger: if(!TriggerHelper.hasRun) { TriggerHelper.hasRun = true; // your logic here }
Static variables reset between transactions but persist within the same transaction.
QWhat is the "One Trigger Per Object" rule? ▼
Best practice: have only one trigger per object. Put all logic in a handler/helper class.

Why? When multiple triggers exist on the same object, Salesforce doesn't guarantee execution order. One trigger could conflict with another.

// Thin trigger — just calls the handler trigger CaseTrigger on Case (before insert, after update) { CaseTriggerHandler.handle(Trigger.operationType); }
QWhat is the Order of Execution in Salesforce? ▼
When a record is saved, Salesforce processes in this order:
  1. System validation (required fields, data type)
  2. Before triggers
  3. Custom validation rules
  4. Record saved to database (not committed)
  5. After triggers
  6. Assignment Rules
  7. Auto-response Rules
  8. Workflow Rules (legacy)
  9. Processes / Flows
  10. Escalation Rules
  11. Database committed (transaction committed)
QCan an Apex Trigger perform a Web Service callout directly? ▼
No. You cannot make a synchronous callout from a trigger — it would block the transaction.

Solution: Use @future(callout=true) or Queueable Apex (implements Database.AllowsCallouts) to make the callout asynchronously.
QWhat are key Governor Limits for triggers? ▼
LimitValue
SOQL queries per transaction100
DML statements per transaction150
Records per DML operation10,000
CPU time per transaction10,000ms (sync), 60,000ms (async)
Heap size6MB (sync), 12MB (async)
QHow do you use addError() in a trigger? ▼
addError() prevents record save and shows an error message to the user — without throwing an exception.

trigger PreventDeleteCase on Case (before delete) { for(Case c : Trigger.old) { if(c.Status == 'Open') { c.addError('Cannot delete an open Case!'); } } }
addError() marks the specific record as failed in bulk operations, allowing other records to succeed if using partial success.
QWhat are Static Variables and how do they behave in bulk uploads? ▼
Static variables persist for the entire transaction (not just one trigger execution). In bulk uploads via Data Loader:

  • Data Loader sends records in batches of 200
  • Each batch is a separate transaction
  • So static variables reset between batches
⚠️ If you use a static Boolean to prevent recursion, it works within one batch but resets for the next batch.

⏱️ Asynchronous Apex

The 4 Async Patterns
QWhen do you use each async type? (Quick Selection Guide) ▼
NeedUse
Simple async task, minimal params@future Method
Complex params, job monitoring, chainingQueueable Apex
Millions of recordsBatch Apex
Time-based / nightly processingScheduled Apex
External API callout from trigger@future(callout=true)
QFuture Methods — rules and limitations? ▼
@future(callout=true) public static void sendToExternalSystem(List<Id> recordIds) { // async logic here }
Rules:
  • Must be static and return void
  • Only primitive parameter types (String, Integer, List<Id>) — no sObjects
  • Cannot call another @future from a @future method
  • No job monitoring / no Job ID
  • Max 50 future calls per transaction
QQueueable Apex — advantages over @future? ▼
Queueable implements the Queueable interface and is enqueued with System.enqueueJob().

Key advantages:
  • ✅ Accepts complex parameters (sObjects, custom types)
  • ✅ Returns a Job ID for monitoring
  • ✅ Supports job chaining (one Queueable can enqueue another)
  • ✅ Supports callouts via Database.AllowsCallouts
💡 Queueable is the modern recommended async pattern for most use cases.
Max chaining depth: 5 levels.
QBatch Apex — how does it work? Key limits? ▼
Batch Apex implements Database.Batchable<SObject> with 3 methods:

  • start() – Returns a QueryLocator or Iterable; collects records to process
  • execute() – Processes each batch; governor limits reset per batch
  • finish() – Runs after all batches complete
LimitValue
Default batch size200 records
Max batch size2,000 records
QueryLocator max records50 million
Database.Stateful – Implement this to preserve variables across batches (for counters/aggregations).
QScheduled Apex — cron syntax? ▼
Implements Schedulable interface. Schedule using:
System.schedule('Job Name', '0 0 2 * * ?', new MyScheduler());
Cron format: Seconds Minutes Hours Day Month DayOfWeek [Year]

  • 0 0 2 * * ? → Every day at 2:00 AM
  • 0 30 8 ? * MON-FRI → 8:30 AM on weekdays
Common pattern: Scheduled Apex calls Batch Apex or Queueable inside execute().
QAsync Apex comparison — all 4 types? ▼
FeatureFutureQueueableBatchScheduled
Complex params❌✅✅✅
Callout support✅✅✅Indirect
Job chaining❌✅ (5 deep)✅Indirect
Large data volume❌❌✅ (50M)Via Batch
Job monitoring❌✅ Job ID✅ Job ID✅ Job ID
QCan a future method call another future method? ▼
No. This is a hard platform limit. Calling a @future method from another @future method results in a runtime exception.

If you need sequential async jobs → use Queueable chaining instead.
QHow do you enable callouts in Batch Apex? ▼
Implement Database.AllowsCallouts on the batch class:
public class MyBatch implements Database.Batchable<SObject>, Database.AllowsCallouts { // Now you can make HTTP callouts in execute() }

⚛️ Lightning Web Components

LWC Architecture & Core Concepts
QWhat are the files in an LWC component? ▼
Every LWC consists of (at most) 4 files:

  • 📄 HTML – Template markup with data bindings and directives (required)
  • 📜 JavaScript – Component logic, state, lifecycle hooks (required)
  • 🎨 CSS – Scoped styles (optional)
  • ⚙️ Meta XML – Exposes component to Salesforce platform (optional but common)
Only HTML and JS are required.
QWhat are the LWC Lifecycle Hooks in order? ▼
HookWhen it runsCommon use
constructor()When component is createdInitialize properties, call super()
connectedCallback()When inserted into DOMFetch data, subscribe to events
renderedCallback()After every renderDOM manipulation
disconnectedCallback()When removed from DOMClean up event listeners
QWhat are the 3 LWC Decorators? ▼
@api – Exposes a public property/method. Enables parent → child communication.
@api recordId;
@track – Makes object properties reactive; LWC detects nested changes and re-renders.
@track account = {};
@wire – Reactive data service. Auto-refreshes when source data changes.
@wire(getRecord, { recordId: '$recordId' }) record;
💡 For simple primitive properties, @track is not needed — LWC is reactive by default for primitives.
Q@wire vs Imperative Apex call — when to use each? ▼
@wire (Reactive):
import getAccounts from '@salesforce/apex/AccountController.getAccounts'; @wire(getAccounts) accounts;
Use when: data should auto-refresh when input changes. Requires cacheable=true in Apex.

Imperative (On-Demand):
getAccounts() .then(result => { this.accounts = result; }) .catch(error => { this.error = error; });
Use when: triggered by user action (button click), when you need to pass data back, or when not cacheable.
QComponent Communication patterns in LWC? ▼
DirectionMechanism
Parent → ChildPass data via @api property or call a child's @api method
Child → ParentDispatch a CustomEvent; parent listens with onsave={handler}
Unrelated componentsLightning Message Service (LMS) — works across LWC, Aura, and Visualforce
QWhat is Lightning Data Service (LDS)? ▼
LDS allows CRUD operations without writing Apex. It:
  • Automatically respects sharing rules and FLS
  • Uses optimized client-side caching
  • Provides: createRecord(), updateRecord(), deleteRecord()
Use LDS when possible — it's faster, more secure, and reduces maintenance vs custom Apex.
Qfor:each vs lwc:if — how do they work in templates? ▼
Iteration:
<template for:each={accounts} for:item="acc"> <p key={acc.Id}>{acc.Name}</p> </template>
Always provide a unique key — Salesforce uses it for efficient re-rendering.

Conditional:
<template lwc:if={showData}>Data Available</template> <template lwc:else>No Data</template>
QWhat does cacheable=true mean in Apex for LWC? ▼
cacheable=true is required on an Apex method when using @wire. It tells Salesforce the method is safe to cache — results may be served from cache.

@AuraEnabled(cacheable=true) public static List<Account> getAccounts() { return [SELECT Id, Name FROM Account]; }
⚠️ Omit cacheable=true for imperative calls. If the method performs DML, it cannot be cached.

🧪 Apex Testing

Core Test Concepts
QWhat is the 75% code coverage requirement? ▼
  • Salesforce requires at least 75% code coverage to deploy Apex to production
  • Individual triggers must have at least 1% coverage (but 75% org-wide is enforced)
  • In sandbox, deployment doesn't require coverage
⚠️ Coverage is measured per class. A test with no assertions but that executes all lines still counts — but assertions are best practice.
QWhat is @isTest, Test.startTest(), Test.stopTest()? ▼
@isTest – Marks a class or method as a test; not counted toward code coverage; not deployed to production.

Test.startTest() – Resets governor limits for code inside the block. Only 1 startTest/stopTest pair per test method.

Test.stopTest() – Ends the test context and forces async operations (like @future) to complete.

@isTest static void testMyMethod() { // Setup data Account a = new Account(Name='Test'); insert a; Test.startTest(); MyClass.doSomething(a.Id); // fresh governor limits here Test.stopTest(); // Assert after stopTest (async has completed) Account result = [SELECT Name FROM Account WHERE Id = :a.Id]; System.assertEquals('Updated', result.Name); }
QWhat is SeeAllData and when should you use it? ▼
By default, test methods use SeeAllData=false — they cannot see existing org data; they must create their own test data.

To allow access to existing data: @isTest(SeeAllData=true)

⚠️ Avoid SeeAllData=true — tests become dependent on live data and can fail unpredictably. It also impacts deployment performance. Always create test data explicitly.
QWhat is a Test Data Factory and why is it best practice? ▼
A Test Data Factory is a utility class (marked @isTest) that creates standardized test records for use across multiple test classes.

Why use it:
  • Centralize data creation — change in one place
  • Reduce code duplication across test classes
  • Maintain consistency in test records
@isTest public class TestDataFactory { public static Account createAccount(String name) { Account a = new Account(Name=name); insert a; return a; } }
QHow do you test @future and async methods? ▼
Wrap the async call in Test.startTest() / Test.stopTest(). When stopTest() is called, Salesforce forces all async operations in the test to complete synchronously.

Test.startTest(); MyFutureClass.doAsyncWork(recordId); // @future call Test.stopTest(); // future method executes HERE // Now assert results System.assertEquals(...)
QHow do you test HTTP callouts? ▼
Salesforce blocks actual HTTP calls during tests. Use the HttpCalloutMock interface:

// 1. Create mock class @isTest global class MockHttpResponse implements HttpCalloutMock { global HTTPResponse respond(HTTPRequest req) { HttpResponse res = new HttpResponse(); res.setBody('{"status":"success"}'); res.setStatusCode(200); return res; } } // 2. In test method Test.setMock(HttpCalloutMock.class, new MockHttpResponse()); Test.startTest(); MyCalloutClass.makeCallout(); Test.stopTest();
QWhat is System.runAs()? Does it enforce FLS? ▼
System.runAs(user) allows you to run a block of code as a specific user — useful for testing sharing rules and user permissions.

User u = [SELECT Id FROM User WHERE Username = 'test@test.com']; System.runAs(u) { // Code executes as this user }
⚠️ System.runAs() does NOT enforce field-level security (FLS) or object-level permissions — it only applies sharing/visibility rules.
QWhy test bulk scenarios (200 records)? ▼
Salesforce triggers receive up to 200 records at once in a single transaction. If your code isn't bulkified:
  • SOQL/DML in loops hit governor limits
  • Real data loads will fail silently or throw errors
@isTest static void testBulk() { List<Case> cases = new List<Case>(); for(Integer i = 0; i < 200; i++) { cases.add(new Case(Subject = 'Test ' + i)); } insert cases; // Triggers the trigger with 200 records }
